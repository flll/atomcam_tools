# 実機ヘルス監査 2026-07-10(稼働 4.5 日・読み取りのみ)

対象: ATOMCam2 実機 10.0.0.228 / ビルド 2.5.20(コミット 9d95ed1 相当、7/6 01:38 起動)。
運用状態: ユーザーが WebUI から RTSP main + WebRTC を有効化し実利用中。連続録画 on。
監査方法: SSH 読み取りのみ(dmesg / top / ps / /proc / atomhack.log / perf_sampler リング 4.4h / 各種ログ)。

## 総合評価

**RTSP + WebRTC + 連続録画 + WebUI を 87MB RAM の単核 MIPS で同時稼働させて 4.5 日無障害**。
軽量化(zram・awk storm 根治・boot 30s)の効果は本番負荷でも持続している。
ただしメモリの余裕はほぼゼロで、SD は自動削除バグにより 96% 定常。

## 悪い点 → 対応

| # | 問題 | 実測根拠 | 対応 |
|---|------|----------|------|
| 1 | **SD 空き床ガード不達**(CIFS 未使用時の early-exit の後ろに実装されていた) | 残 602MB(<1GB)でも削除記録ゼロ。サンドボックス再現で df 呼び出し 0 回 | **修正済 349f562**(要デプロイ) |
| 2 | **crond 二重起動** → 全 cron ジョブが毎回 2 回実行 | crond 2 プロセス(-f / -l 8)、daily mem が毎回 2 行 | **修正済 ab82356**(要デプロイ) |
| 3 | **非表示タブでも JPEG/ISP ポーリング継続**(カメラへ最大 2req/s の無駄) | コードレビューで特定 | **修正済 a9401d8**(PiP 中は継続、要デプロイ) |
| 4 | メモリ余裕ゼロ + **緩い漸増**(used 57.9→61.0MB、+1MB/日 弱) | atomhack.log daily mem 7/7→7/10。4.4h 窓ではプロセス RSS フラット=遅いリークor断続確保 | **観測強化済 38c98d5**: 日次でプロセス別 RSS を記録(要デプロイ)。犯人特定は数日分のデータ後 |
| 5 | SD 96% 定常(602MB free) | 純正 FW が record を約 3 日で自前ローテしギリギリ延命 | #1 のデプロイで 1GB 床を確保。恒久対応は自動削除 on の検討(帰宅後判断) |
| 6 | PERF_SAMPLER=on 切り忘れ | CPU 累計 11 分/4.5 日(≈0.17%)、tmpfs 2MB | 実害小。今回の解析に有効だったため on 継続も合理的(判断委ね) |
| 7 | 純正ログ log_*_s.txt が難読化 | 中身の監査不能 | 仕様(対応なし) |

## スペック所感(「この機種でどこまでやれるか」)

- **CPU**: まだ 60% idle。もう1系統の配信(HEVC)を足すと苦しい可能性
- **RAM が最初に尽きる**: RTSP+WebRTC 有効化で used 32MB→61MB。フル構成の常用は
  zram 前提でぎりぎり成立。HomeKit 等の追加は非推奨
- **SD I/O**: 連続録画 ~4GB/日。低速カードだと mmcqd がボトルネック化する構成
